I'll do anything to avoid learning tabletop game battle rules. Apparently "anything" includes writing a formal specification of D&D 5e combat.

D&D is a narrative tabletop game — one of many, albeit the most popular. Players act as fictional characters within a shared story, guided by mechanical rules.

The rules create tactically interesting battles. I find them hard to learn and remember — I value the storytelling part more. So I made a state machine to run the battle rules for me.

## What I built

A [Quint](https://github.com/informalsystems/quint) formal spec of the 2024 SRD[^srd] combat rules — conditions, action economy, grappling, death saves, spellcasting — modeled as a single-actor state machine. An [XState v5](https://xstate.js.org/) implementation mirrors it exactly.

Model-based conformance testing ([`@firfi/quint-connect`](https://github.com/dearlordylord/quint-connect-ts)) replays random traces against both and compares state field-by-field. A thin React UI sits on top.

Each layer checks the one above it. The UI can't render a state the machine can't produce. The machine can't produce a state the spec forbids.

The MBT traces are seed-deterministic: same seed, same trace, fully reproducible. I used the same approach for [property-based testing of temporal graph storage](/blog/graph-property-based): deterministic seeds let you reproduce any failure and run thousands of scenarios.

Here, the formal spec replaces hand-written properties as the oracle[^oracle].

## What is Quint

[Quint](https://github.com/informalsystems/quint) is an executable specification language based on the temporal logic of actions (TLA) — the same foundation as [TLA+](https://lamport.azurewebsites.net/tla/tla.html), but with syntax familiar to engineers.

You write pure functions that describe state transitions. Quint model-checks them: exhaustively explores every reachable state and verifies your invariants hold. Built by [Informal Systems](https://informal.systems/), the team behind Cosmos blockchain verification.

## Crowdsourced spec testing

1,000 machine-checked assertions distilled so far from a corpus of 12,700 Reddit and StackExchange Q&A arguments between dedicated D&D players. An LLM translates them into Quint. Surviving assertions validate the spec against community domain knowledge. The corpus has a lot more to give.

One example. Grapple a creature, knock it prone. Its speed drops to 0. Standing from prone costs half your speed. Half of zero is zero. It's stuck on the ground forever.

An LLM turns a Reddit discussion into a formal assertion, then the spec confirms it:

```ts
// qa_generated.qnt
run qa_prone_and_grappled_cannot_stand = {
  val target = freshCreature(30)
  val grappled = pGrapple(Medium, Medium, target, true, true).targetState
  val grappledAndProne = pApplyCondition(grappled, CProne)
  val t = pStartTurn(TEST_CONFIG, grappledAndProne, Unarmored, 0, false, false)
  val tAfterStandAttempt = pStandFromProne(t)
  assert(t.effectiveSpeed == 0 and tAfterStandAttempt.movementRemaining == t.movementRemaining)
}
```

The same pipeline can be fed game session recordings — another source of real-world rule interactions to validate the spec against.
